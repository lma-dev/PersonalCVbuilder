# CLAUDE.md

## Tech Stack

- **Framework:** Next.js 16 (App Router) + React 19 + TypeScript 5
- **Database:** PostgreSQL 16 + Prisma 7 (with `@prisma/adapter-pg`)
- **Auth:** NextAuth v4 (Credentials provider, JWT strategy)
- **Styling:** Tailwind CSS v4 + shadcn/ui (new-york style, lucide icons)
- **State:** Jotai (client) + TanStack React Query v5 (server)
- **Validation:** Zod v4
- **Package Manager:** pnpm

## Commands

```bash
pnpm dev              # Start dev server
pnpm build            # Production build
pnpm start            # Start production server
pnpm lint             # Run ESLint
pnpm test             # Run vitest
pnpm test:watch       # Run vitest in watch mode
pnpm db:seed          # Seed data
pnpm prisma migrate dev      # Create + apply migration
pnpm prisma migrate deploy   # Apply pending migrations (production)

```

## Architecture Patterns

### Feature Module Pattern

Every feature follows: `domain/ → data/ → schema.ts → api-client.ts → use-*.ts → components/`

- **domain/** — Pure business logic, no framework dependencies
- **data/** — Prisma queries only, called by domain or API routes
- **schema.ts** — Zod schemas for request validation
- **api-client.ts** — Frontend `fetch()` wrapper for the feature's API
- **use-*.ts** — React Query hooks consuming api-client
- **atoms.ts** — Jotai atoms for client-side UI state

### API Route Pattern

```typescript
// app/api/[resource]/route.ts
import { requireAuth } from "@/lib/api-auth";

export async function GET(req: Request) {
  const session = await requireAuth();  // throws 401
  // ... handle request
  return Response.json(data);
}
```

### Database Access

- Prisma client: `import { prisma } from "@/lib/prisma"`
- Generated client output: `app/generated/prisma/` (configured in schema.prisma)
- Uses `@prisma/adapter-pg` for connection pooling
- Soft delete pattern: `deletedAt` column (NULL = active)

### Auth

- **API auth** (`lib/api-auth.ts`): `requireAuth()` for API route protection
- Always use `requireAuth()` in API routes, never trust client-side auth checks

## Coding Conventions

- **Path alias:** `@/` maps to project root
- **Imports:** Use `@/` prefix (e.g., `@/lib/prisma`, `@/features/auth/domain`)
- **Components:** PascalCase filenames, `"use client"` directive for interactive components
- **API routes:** RESTful, grouped by resource under `app/api/`
- **Styling:** Tailwind utility classes + `cn()` from `@/lib/utils` for conditional classes
- **Toasts:** `sonner` — use `toast.success()`, `toast.error()`
- **Icons:** `lucide-react` only
- **Forms:** Zod schema → validate in API route, client-side validation in components
- **Error classes:** Each feature has its own `error.ts` with a custom Error subclass
- **Export formats:** Excel (exceljs), PDF (pdfkit), JSON — see `features/*/export/`

## Important Notes

- **DO NOT edit** `app/generated/prisma/` — auto-generated by `prisma generate`
- **Soft delete** — Use `deletedAt` instead of hard delete

## Security Guidelines (MUST FOLLOW)

Security is a **core requirement**, not optional. All features must follow these rules.

---

### 🔐 Authentication

- Always use `requireAuth()` in protected API routes
- Never trust client-side authentication state
- Use secure cookies (`httpOnly`, `secure`, `sameSite=strict`)
- Ensure JWT has expiration (short-lived tokens)
- Implement account lockout after multiple failed login attempts
- Enforce strong password policy (min length, complexity)

---

### 🔑 Authorization (CRITICAL)

- Implement role-based access control (RBAC)
- Validate permissions in EVERY API route
- Never trust user role from frontend
- Protect admin routes strictly
- Always verify resource ownership (user can only access their own data)

---

### 🧾 Input Validation (Zod)

- Validate ALL API inputs using Zod
- Reject unexpected fields
- Limit input size (prevent large payload attacks)
- Validate query params, body, and headers
- Never pass raw user input directly to database

---

### 🌐 API Security

- Implement rate limiting (per IP / per user)
- Configure CORS (allow only trusted domains)
- Use HTTPS only in production
- Prevent brute-force attacks (especially login endpoints)
- Add request timeout handling

---

### 🧱 Security Headers

Apply via Next.js middleware:

- Content-Security-Policy (CSP)
- X-Frame-Options: DENY
- X-Content-Type-Options: nosniff
- Referrer-Policy
- Strict-Transport-Security (HSTS)

---

### 🛡️ CSRF & XSS Protection

- Protect all POST/PUT/DELETE endpoints from CSRF
- Escape all user-generated content before rendering
- Avoid `dangerouslySetInnerHTML`
- Use trusted UI components (shadcn/ui)

---

### 🗄️ Database Security

- Use Prisma ORM (avoid raw SQL when possible)
- Validate all inputs before database queries
- Restrict database user permissions
- Use soft delete (`deletedAt`) instead of hard delete
- Implement regular backups
- Encrypt sensitive fields if needed

---

### 🔑 Secrets Management

- Store secrets in `.env` files
- Never expose secrets to frontend
- Use different environments (dev/staging/prod)
- Rotate secrets periodically
- Protect API keys strictly

---

### 📊 Logging & Monitoring

- Log all critical actions (create/update/delete)
- Log failed login attempts
- Log suspicious activity
- Monitor API errors
- Use `logAction()` for audit trails

---

### 🚨 Abuse Protection

- Rate limit sensitive endpoints (login, API)
- Prevent spam and abuse
- Add CAPTCHA for public forms (optional)
- Detect unusual activity patterns

---

### 📁 File Upload Security (if applicable)

- Validate file type (MIME)
- Limit file size
- Rename uploaded files (avoid original filenames)
- Store files outside public directory
- Scan files if necessary (advanced)

---

### 🧠 Business Logic Security

- Enforce business rules in backend (NOT frontend)
- Prevent privilege escalation
- Validate all state transitions
- Handle race conditions properly

---

### ⚙️ Deployment Security

- Use HTTPS with valid SSL certificate
- Disable debug mode in production
- Secure hosting environment (Vercel/AWS/etc.)
- Protect admin endpoints
- Configure firewall / WAF (optional)

---

### 📦 Dependency Security

- Regularly run `pnpm audit`
- Keep dependencies up to date
- Remove unused packages
- Monitor known vulnerabilities

---

### ❗ General Principles

- Never trust user input
- Always validate and sanitize data
- Apply security in layers (defense in depth)
- Security must be considered from the start, not added later
